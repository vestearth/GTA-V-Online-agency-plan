# Dashboard Data Map

The static dashboard is a manual snapshot surface. It is not generated by a build step, script, or backend today. Update values by reviewing the current profile, the latest weekly payload, and the standard weekly reports before editing `dashboard.html`.

| Dashboard field | Source | Update rule |
|---|---|---|
| Player name / platform | `data/player_profile.json` | Copy the current profile identity into the dashboard header when the profile changes. |
| Week label | latest `data/weekly_planning_*.json` | Use the most recent weekly payload and keep the date range aligned with that week. |
| Bankroll | `data/player_profile.json` -> `bankroll.cash_on_hand` | Manual snapshot; round or format for display only after checking the current numeric value. |
| Weekly spend cap | `data/player_profile.json` -> `bankroll.max_spend_this_week` | Manual snapshot from the current profile. |
| Owned Major Assets | `data/player_profile.json` -> `owned_assets.properties` | Count manually from the current array or replace with a generated value later if automation is introduced. |
| Missing Major Assets | `data/player_profile.json` -> `owned_assets.missing_properties` | Use the current list length; if the notes field conflicts with the arrays, prefer the arrays and explain the inconsistency in the dashboard or reports. |
| Asset Overview table | `data/player_profile.json`, latest `data/weekly_planning_*.json`, and weekly reports in `reports/` | Curated manually. Use profile ownership as the source of truth, then add weekly priority notes from the latest plan. |
| Upgrade Checklist | `data/player_profile.json` -> `owned_assets.upgrades` | Manual snapshot from the current upgrade flags. |
| Current Focus | latest `data/weekly_planning_*.json` plus `reports/weekly_master_plan_<week_id>.md` | Manual summary of the top recommended weekly loop. |
| Next Claim / Buy | `reports/weekly_master_plan_<week_id>.md` and `reports/event_master_plan_<week_id>.md` | Manual summary of the highest-priority claim or purchase action after ownership and readiness checks. |
| Weekly Action Plan | `reports/weekly_master_plan_<week_id>.md` | Manual summary of the ordered action queue from the latest master plan. |
| ROI / Passive Income | `reports/weekly_master_plan_<week_id>_income_scenarios.md` and `reports/weekly_master_plan_<week_id>.md` | Manual summary of the best active loop and passive filler notes for the current week. |
| What to Buy / Ignore | `reports/weekly_master_plan_<week_id>.md` and `reports/event_master_plan_<week_id>.md` | Manual summary of recommended buy, claim, skip, and ignore rulings. |
| Decision Log | weekly reports in `reports/` plus user-confirmed history | Advisory only. Do not imply historical decisions were executed unless reports or the user confirm it. |
| Data source note | `data/player_profile.json`, latest `data/weekly_planning_*.json`, `reports/` | Keep the note visible in `dashboard.html` and update it only if the dashboard stops being manually maintained. |
| Last reviewed | dashboard maintainer edit date | Update on every manual dashboard refresh using `YYYY-MM-DD`. |
| Data status | dashboard maintainer judgment | Use `Manual / Needs sync` until a generator exists and the page is refreshed automatically. |

## Current Weekly Source Convention

For the current dashboard baseline, treat these files as the canonical weekly bundle when they exist for the same week:

- `data/weekly_planning_<week_id>.json`
- `reports/weekly_master_plan_<week_id>.md`
- `reports/weekly_master_plan_<week_id>_income_scenarios.md`
- `reports/event_master_plan_<week_id>.md`

Example current week in this repository: `2026_w22`.

## Future Roadmap

- A future `scripts/generate_dashboard.py` may read `data/player_profile.json`, the latest `data/weekly_planning_*.json`, and the latest standard weekly report files to rewrite `dashboard.html`.
- Keep that generator optional until manual dashboard upkeep becomes annoying enough to justify automation.
- Do not move to React, Vue, Next, Tailwind, or another dynamic dashboard stack unless the static workflow becomes painful and there is a concrete need for filtering, sorting, live data loading, or repeated weekly generation.
